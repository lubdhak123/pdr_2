## 1. PROJECT OVERVIEW

PDR (Proxy Data Rating) is an **alternative credit scoring system** for **Indian MSMEs and NTC (New-To-Credit) borrowers** who lack traditional credit bureau history. Instead of CIBIL scores, PDR uses:

- **Bank statement analysis** (via Account Aggregator framework)
- **GST filing records**
- **Telecom data** (SIM vintage, recharges)
- **Utility payment history**
- **Middleman data** (supplier invoices, BC agent cash deposits)

**Key innovation:** Cash-only MSMEs with NO bank account can be scored using the **Middleman Data Path** — where 5 trusted middlemen (supplier, GST portal, telecom, utility, BC agent) provide data independently.

**Target users:**
- 30 million cash-only MSMEs in India
- NTC individuals with no credit history
- Gig workers, seasonal businesses, micro-retailers

### 3.3 Model Performance

```

AUC-ROC       : 0.9378

KS Statistic  : 0.7262
Calibration   : max error 0.072
```

### 3.4 CRITICAL: monthly_income is transaction-derived

`monthly_income` = `sum(transaction credits) / months_span` — computed entirely from bank statement transactions, NOT from the "Annual Income" form field. Changing the form field changes DTI display only. To change the income used in scoring, the transactions in `demo_users.json` must change.

### 4.4 Fraud Hard-Override (in scorer.py)

After model scoring, two hard overrides fire regardless:
```python
if features.get('p2p_circular_loop_flag', 0) == 1:
    pd_score = max(pd_score, 0.85)   # forces Grade E
if features.get('identity_device_mismatch', 0) == 1:
    pd_score = max(pd_score, 0.75)   # forces Grade D/E
```

### 4.5 Model Routing (NTC vs MSME)

```python
is_ntc = (
    profile.get('business_vintage_months', 0) == 0
    or profile.get('business_type') == 'Individual / NTC'
)
```

NTC model = `ntc_credit_model.pkl` (CalibratedClassifierCV wrapping XGBoost)
MSME model = `xgb_msme_raw.pkl` (raw XGBoost — avoids PlattCalibratedModel pickle error)

SHAP for NTC: uses `ntc_model.calibrated_classifiers_[0].estimator` to get the raw booster.

## 5. PRE-LAYER RULE ENGINE (`pre_layer.py`)

3-tier rule engine. Returns `(grade, outcome, reason)` tuple OR `None` (→ model runs).

### Type 1 — Hard Rejection
- P2P circular flow + bounced transactions
- GST variance > 150%
- 5+ bounces
- 3+ minimum balance violations
- >80% cash dependency + bounces

### Type 2 — Edge Case Protection (fast-track B)
- NRI/platform worker with low cash dependency
- Seasonal business with legitimate GST filings
- New business with strong telecom vintage + clean GST
- High-value stable borrower with minor utility delays

### Type 3 — Manual Review Triggers
- High customer concentration (>85% from 1-2 clients)
- Region risk + cashflow volatility
- Income drop > 30% with low buffer

| Method | Endpoint | Purpose |
|--------|----------|---------|
| `POST /score` | Direct scoring (transactions + profile + GST) |
| `GET /health` | Health check |

### POST /score/declined (B2B Bank Referral Endpoint)

```python
class DeclinedRequest(BaseModel):
    applicant_name: str
    bank_rejection_reason: str = 'Insufficient credit history'
    monthly_income: Optional[float] = None
    loan_amount_requested: Optional[float] = None
    user_profile: Optional[Dict[str, Any]] = None
    transactions: Optional[list] = None
    gst_data: Optional[Dict[str, Any]] = None
```

Logic:
- If transactions provided → run full PDR scoring
- If PDR grades A/B/C → return loan offer (PDR overrules bank decline)
- If PDR grades D/E → return ALTERNATIVE_PRODUCTS
- If no data → return generic alternative products

### 8.3 Transaction Forensics (`TransactionForensics.jsx`)

NEW component (Phase 11). Renders for ALL profiles with transactions.

**Toggle button in Results.jsx:**
- Button: "View Transaction Forensics" / "Hide Transaction Forensics"
- Shows a red "Signals detected" badge if active_flags contains any fraud/risk flags
- State: `const [showForensics, setShowForensics] = useState(false)`

**Component logic:**
- Accepts: `{ transactions, activeFlags }`
- All React hooks (useRef, useEffect) declared BEFORE any early return (Rules of Hooks fix — this was the bug that caused silent crash in StrictMode)
- `annotated = annotate(transactions)` — runs client-side detection:
  - **Circular loop:** credit from entity X → debit to same X within 3 days, ≥78% amount match. Uses `extractEntity()` to strip UPI prefixes ("UPI TRANSFER FROM/TO", "NEFT", etc.)
  - **Bounce:** narration matches `/bounce|chg/i`
  - **Round number:** `|amount| ≥ ₹10,000 AND divisible by ₹1,000`
  - **Income spike:** credit > 3.5× average monthly credit

**Rendered elements:**
1. Summary cards (circular count, bounce count, round count, spike count) — only cards with count > 0 shown
2. Chart.js stacked bar chart (Transaction Timeline — Flagged vs Normal): green=normal credit, grey=normal debit, red=circular, purple=spike, orange=bounce. Uses `window.Chart` (loaded via CDN in index.html)
3. Annotated transaction table: color-coded rows, flag badges per transaction
4. Header shows "X flagged transactions" (red badge) OR "All clean" (green badge)

**FRAUD_FLAGS checked for "Signals detected" badge:**

## 9. INFERENCE PIPELINE (FULL FLOW)

```
POST /score (or GET /demo/{id})
  → feature_engine.py: compute_features(transactions, profile, gst_data)
       ★ monthly_income = sum(CREDIT amounts) / months_span  ← from TRANSACTIONS, not form
       → returns ~50 feature dict
  → pre_layer.py: apply_pre_layer(features)
       → if rule fires: return (grade, outcome, reason) — model skipped
       → else: return None
  → routing: is_ntc? → ntc_model else msme_model
  → scorer._to_row(): fill missing features with 0, add MSME engineered cols
  → model.predict_proba() → pd_score
  → shap.TreeExplainer().shap_values() → top 5 reasons
  → fraud hard-override (p2p_circular_loop → ≥0.85, identity_mismatch → ≥0.75)
  → _grade(pd_score) → grade, outcome
  → _loan_offer(grade, features) → loan offer or alternative products
  → return full result dict
```

## 14. KEY DESIGN DECISIONS

1. **70/30 behavioral-demographic split:** TARGET = 0.70 × behavioral_score + 0.30 × demographic_risk.
2. **Single model for both paths:** AA and Middleman paths use the SAME XGBoost model.
3. **Confidence-adjusted thresholds:** Fewer middleman sources → tighter APPROVE bars.
4. **Pre-layer overrides ML:** Hard fraud signals bypass the model entirely.
5. **monthly_income from transactions:** Income is always derived from bank credits, never self-reported form fields — prevents gaming.
6. **Farouk is a permanent fraud demo:** MSME_002 always scores E. His transaction pattern (₹1L circular loops with Shyam Enterprises) is hardcoded in demo_users.json to demonstrate the fraud detection pipeline.
7. **TransactionForensics is universal:** The component uses client-side detection (no extra API call) and auto-detects fraud signals in any profile's transactions. Shown via toggle button so it doesn't clutter clean profiles.
