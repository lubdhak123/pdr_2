## 1. PROJECT OVERVIEW

PDR (Proxy Data Rating) is an **alternative credit scoring system** for **Indian MSMEs and NTC (New-To-Credit) borrowers** who lack traditional credit bureau history. Instead of CIBIL scores, PDR uses:

- **Bank statement analysis** (via Account Aggregator framework)
- **GST filing records**
- **Telecom data** (SIM vintage, recharges)
- **Utility payment history**
- **Middleman data** (supplier invoices, BC agent cash deposits)

**Key innovation:** Cash-only MSMEs with NO bank account can be scored using the **Middleman Data Path** — where 5 trusted middlemen (supplier, GST portal, telecom, utility, BC agent) provide data independently.

**Target users:**
- 30 million cash-only MSMEs in India
- NTC individuals with no credit history
- Gig workers, seasonal businesses, micro-retailers

### 3.4 CRITICAL: monthly_income is transaction-derived

`monthly_income` = `sum(transaction credits) / months_span` — computed entirely from bank statement transactions, NOT from the "Annual Income" form field. Changing the form field changes DTI display only. To change the income used in scoring, the transactions in `demo_users.json` must change.

## 5. PRE-LAYER RULE ENGINE (`pre_layer.py`)

3-tier rule engine. Returns `(grade, outcome, reason)` tuple OR `None` (→ model runs).

### Type 1 — Hard Rejection
- P2P circular flow + bounced transactions
- GST variance > 150%
- 5+ bounces
- 3+ minimum balance violations
- >80% cash dependency + bounces

### Type 2 — Edge Case Protection (fast-track B)
- NRI/platform worker with low cash dependency
- Seasonal business with legitimate GST filings
- New business with strong telecom vintage + clean GST
- High-value stable borrower with minor utility delays

### Type 3 — Manual Review Triggers
- High customer concentration (>85% from 1-2 clients)
- Region risk + cashflow volatility
- Income drop > 30% with low buffer

### 8.3 Transaction Forensics (`TransactionForensics.jsx`)

NEW component (Phase 11). Renders for ALL profiles with transactions.

**Toggle button in Results.jsx:**
- Button: "View Transaction Forensics" / "Hide Transaction Forensics"
- Shows a red "Signals detected" badge if active_flags contains any fraud/risk flags
- State: `const [showForensics, setShowForensics] = useState(false)`

**Component logic:**
- Accepts: `{ transactions, activeFlags }`
- All React hooks (useRef, useEffect) declared BEFORE any early return (Rules of Hooks fix — this was the bug that caused silent crash in StrictMode)
- `annotated = annotate(transactions)` — runs client-side detection:
  - **Circular loop:** credit from entity X → debit to same X within 3 days, ≥78% amount match. Uses `extractEntity()` to strip UPI prefixes ("UPI TRANSFER FROM/TO", "NEFT", etc.)
  - **Bounce:** narration matches `/bounce|chg/i`
  - **Round number:** `|amount| ≥ ₹10,000 AND divisible by ₹1,000`
  - **Income spike:** credit > 3.5× average monthly credit

**Rendered elements:**
1. Summary cards (circular count, bounce count, round count, spike count) — only cards with count > 0 shown
2. Chart.js stacked bar chart (Transaction Timeline — Flagged vs Normal): green=normal credit, grey=normal debit, red=circular, purple=spike, orange=bounce. Uses `window.Chart` (loaded via CDN in index.html)
3. Annotated transaction table: color-coded rows, flag badges per transaction
4. Header shows "X flagged transactions" (red badge) OR "All clean" (green badge)

## 9. INFERENCE PIPELINE (FULL FLOW)

```
POST /score (or GET /demo/{id})
  → feature_engine.py: compute_features(transactions, profile, gst_data)
       ★ monthly_income = sum(CREDIT amounts) / months_span  ← from TRANSACTIONS, not form
       → returns ~50 feature dict
  → pre_layer.py: apply_pre_layer(features)
       → if rule fires: return (grade, outcome, reason) — model skipped
       → else: return None
  → routing: is_ntc? → ntc_model else msme_model
  → scorer._to_row(): fill missing features with 0, add MSME engineered cols
  → model.predict_proba() → pd_score
  → shap.TreeExplainer().shap_values() → top 5 reasons
  → fraud hard-override (p2p_circular_loop → ≥0.85, identity_mismatch → ≥0.75)
  → _grade(pd_score) → grade, outcome
  → _loan_offer(grade, features) → loan offer or alternative products
  → return full result dict
```

## 14. KEY DESIGN DECISIONS

1. **70/30 behavioral-demographic split:** TARGET = 0.70 × behavioral_score + 0.30 × demographic_risk.
2. **Single model for both paths:** AA and Middleman paths use the SAME XGBoost model.
3. **Confidence-adjusted thresholds:** Fewer middleman sources → tighter APPROVE bars.
4. **Pre-layer overrides ML:** Hard fraud signals bypass the model entirely.
5. **monthly_income from transactions:** Income is always derived from bank credits, never self-reported form fields — prevents gaming.
6. **Farouk is a permanent fraud demo:** MSME_002 always scores E. His transaction pattern (₹1L circular loops with Shyam Enterprises) is hardcoded in demo_users.json to demonstrate the fraud detection pipeline.
7. **TransactionForensics is universal:** The component uses client-side detection (no extra API call) and auto-detects fraud signals in any profile's transactions. Shown via toggle button so it doesn't clutter clean profiles.

## 17. SURPRISE ELEMENT — AI CREDIT ANALYST CHATBOT (Phase 7)

> **Status as of 2026-03-28:** Steps 1–4 complete. Step 5 (Ollama wire-up + input loop) remaining.

### 17.1 What It Is

A chatbot that lets a **loan officer** query, interrogate, compare, and generate decision letters
for applicants in plain English. Powered by a **local LLM via Ollama** (Mistral 7B or Llama3 8B).
No web framework — runs in a notebook `input()` loop.

Barclays requirement: Use locally hosted LLMs. Claude API is optional via `USE_CLAUDE=true` env var.

### 17.2 Architecture

```
Loan Officer (plain English)
        ↓
chatbot_router.py      → classify query type + extract applicant IDs
        ↓
context_layer.py       → fetch pre-computed applicant card from SQLite
        ↓
chatbot_context.py     → enrich card + build grounded LLM prompt pair
        ↓
Ollama API             → local LLM generates response
        ↓
credit_analyst_chatbot.py → input() loop + display
```

### 17.3 Files Built (all in project root)

#### `context_layer.py` — SQLite Database Layer
- **4 tables:**
  - `applicant_cards` — core card (grade, PD, decision, primary reason, soft-delete)
  - `applicant_features` — normalized 44 features (one row per feature, upsertable)
  - `applicant_shap_explanations` — top-5 SHAP factors with rank/direction/impact
  - `applicant_loan_offers` — rate range, max loan, tenures, recommended product
- **Indexes on:** grade, outcome, score_date, is_deleted
- **Public API:**
  ```python
  init_database(db_path)
  save_applicant_card(db, result, applicant_id, name, city, business_type) → int
  fetch_applicant_card(db, applicant_id) → dict
  search_applicants(db, filters) → list[dict]
  delete_applicant_card(db, applicant_id) → bool  # soft delete
  bulk_save_applicant_cards(db, records) → list[int]
  get_grade_distribution(db) → dict
  ```
- **Input:** full `score_user()` result dict
- **Output of fetch:** `{applicant_id, name, grade, decision, default_probability, primary_reason, top_shap_factors, all_features, loan_offer, alternative_products}`

#### `chatbot_router.py` — Query Classifier
- **7 query types** (enum: `QueryType`):
  - `LOOKUP` — "Pull up applicant ntc_001"
  - `EXPLANATION` — "Why was msme_003 rejected?"
  - `COMPARISON` — "Compare ntc_001 and ntc_002"
  - `SCENARIO` — "What if ntc_001 had 20% more income?"
  - `DECISION_LETTER` — "Generate a rejection letter for msme_003"
  - `RISK_ASSESSMENT` — "What's the biggest risk for ntc_001?"
  - `AGGREGATE` — "How many applicants were rejected?"
- **Returns `RoutedQuery(query_type, applicant_ids, parameters, raw_message, confidence)`**
- Applicant ID patterns: `ntc_001`, `msme_003`, `app_12345`, `#4532`
- Parameter extraction per type:
  - SCENARIO → `magnitude_pct`, `change_direction`, `target_feature`
  - DECISION_LETTER → `letter_type` (rejection/approval/review/auto)
  - AGGREGATE → `decision`, `grade`, `risk_band`, `limit`, `order`
  - EXPLANATION → `focus` (rejection_reason/approval_reason/risk_factors/general)

#### `chatbot_context.py` — Context Fetcher + Prompt Builder (Steps 3 & 4)

**Step 3: `fetch_applicant_context(db_path, applicant_id)` → enriched dict**
- Adds to raw DB card:
  - `enriched_shap` — each factor gets human label (e.g. "Reliance on cash withdrawals"), raw feature value, threshold note ("below 50% is preferred"), and `⚠ RISK`/`✓ STRENGTH` tag
  - `red_flags` — auto-detects: 3+ bounces, P2P circular loop, GST variance >150%, cash >80%, turnover spike, device mismatch
  - `key_features_plain` — priority features formatted as "82%", "1.3 years", "3/12 months"
  - `loan_offer_text` — "Eligible — up to ₹500,000 at 11–14% p.a. over [12,24,36] months"
  - `risk_band` — derived: LOW <20%, MEDIUM 20–45%, HIGH >45%

**Step 4: `build_prompt(routed, db_path)` → `(system_prompt, user_prompt)`**
- 7 templates, one per query type:

| Template | LLM task instruction |
|---|---|
| LOOKUP | 3–4 sentence profile summary |
| EXPLANATION | Cite top 2–3 SHAP drivers with actual values |
| COMPARISON | Side-by-side of two full card blocks |
| SCENARIO | Directional reasoning from SHAP + threshold math |
| DECISION_LETTER | Formal letter (forbidden words: SHAP, XGBoost, model, algorithm, feature) |
| RISK_ASSESSMENT | Primary risk + 2 others + 1 protective factor |
| AGGREGATE | Portfolio SQL results with grade distribution injected |

- Shared system prompt enforces: only use provided data, cite actual values, max 250 words, no model internals revealed

**Feature label dictionary** — all 44 features mapped to plain English (e.g. `bounced_transaction_count` → "Number of bounced/failed transactions")

**Decision threshold reminder** injected in SCENARIO prompts:
```
APPROVE if P(default) < 35% | MANUAL_REVIEW if 35–55% | REJECT if > 55%
```

### 17.7 Ollama API Call (for Step 5)

```python
import requests, os

OLLAMA_URL   = "http://localhost:11434/api/generate"
OLLAMA_MODEL = os.environ.get("OLLAMA_MODEL", "mistral")

def call_ollama(system_prompt: str, user_prompt: str) -> str:
    full_prompt = f"[SYSTEM]\n{system_prompt}\n\n[USER]\n{user_prompt}"
    resp = requests.post(OLLAMA_URL, json={
        "model": OLLAMA_MODEL, "prompt": full_prompt, "stream": False
    }, timeout=120)
    resp.raise_for_status()
    return resp.json()["response"]

def call_llm(system_prompt: str, user_prompt: str) -> str:
    if os.environ.get("USE_CLAUDE","").lower() == "true":
        import anthropic
        client = anthropic.Anthropic(api_key=os.environ["ANTHROPIC_API_KEY"])
        msg = client.messages.create(
            model="claude-sonnet-4-5", max_tokens=1024,
            system=system_prompt,
            messages=[{"role": "user", "content": user_prompt}]
        )
        return msg.content[0].text
    return call_ollama(system_prompt, user_prompt)
```

### 17.8 Main Chat Loop Template (Step 5)

```python
from chatbot_router import route_query
from chatbot_context import build_prompt

DB = "applicant_cards.db"

def main():
    print("=" * 60)
    print("  PDR Credit Analyst Chatbot — Ollama (Mistral)")
    print("  Type 'exit' to quit")
    print("=" * 60)
    while True:
        query = input("\nLoan Officer > ").strip()
        if not query: continue
        if query.lower() in ("exit", "quit"): break
        try:
            routed = route_query(query)
            sys_p, usr_p = build_prompt(routed, DB)
            response = call_llm(sys_p, usr_p)
            print(f"\nAnalyst > {response}")
            print("-" * 60)
        except FileNotFoundError:
            print("ERROR: applicant_cards.db not found. Run build_applicant_cards.py first.")
        except requests.exceptions.ConnectionError:
            print("ERROR: Ollama not running. Start with: ollama serve")
        except Exception as e:
            print(f"ERROR: {e}")

if __name__ == "__main__":
    main()
```
